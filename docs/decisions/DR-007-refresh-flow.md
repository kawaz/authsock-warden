# DR-007: 統一 refresh フロー

- **Status**: Active
- **Date**: 2026-04-02

## 背景
鍵のタイムアウトを手動でリフレッシュしたい場面がある:
- 離席前にタイマーを延長しておきたい
- リモートからAI agentに長時間作業させる前にリフレッシュしたい

ローカル（TouchID可能）とリモート（TouchID不可）で別のコマンドやフローにしたくない。ユーザーは今の環境がローカルかリモートかを意識せずに同じ操作をしたい。

また、refresh の認証は warden 自身が行う独立した認証であり、op item get の成功を認証としてカウントしてはならない。op CLI の認証はあくまで1Passwordへのアクセス認証であり、warden の「この人が今ここにいる」という本人確認とは別物。

## 検討した選択肢

### A. ローカル/リモートで明示的に分岐
`refresh --local`（TouchID）と `refresh --remote`（代替認証）を用意する。

- **利点**: 明示的で予測しやすい
- **欠点**: ユーザーが環境を判断する必要がある。自動化しにくい

### B. op item get を先に試す楽観的フォールバック
まず op item get を試み（TouchIDやopキャッシュで通る可能性）、失敗したら代替認証にフォールバック。

- **利点**: opキャッシュが効けば高速
- **欠点**: op CLI の認証成功を warden の本人確認として扱ってしまう。認証の責任が曖昧になる

### C. warden 独自認証を必須とする統一フロー
warden が設定された認証方式（re-auth command 等）で認証。認証後にオプショナルで op item get を試みる。

- **利点**: 認証の責任が warden にある。command で任意の認証フローを外部委譲可能
- **欠点**: 認証手段の実装・運用がユーザー側の責任になる

## 決定
**C. warden 独自認証を必須とする統一フロー。**

```
refresh:
  1. warden 独自の認証（必須）
     設定された認証方式で認証:
     - re-auth command: 外部スクリプト（TouchIDラッパー、Passkey等）
     → 認証成功: ステップ2へ
     → 認証失敗: エラー
  2. 認証成功後の処理
     - op item get を試みる（ベストエフォート、鍵更新可能なら更新）
     - タイマーリセット
```

## 理由
warden はセキュリティツールであり、「この人が今ここにいる」という本人確認は warden 自身の責任で行うべき。op item get の成功は1Passwordへのアクセス認証に過ぎず、warden の本人確認とは別物。

re-auth command で外部委譲することで:
- ユーザーが自分の環境に最適な認証フローを自由に構築可能（TouchIDラッパー、Passkey、Push通知等）
- ローカル/リモートの判定は command 側の責任。warden は結果（exit code）だけを見る
- ユーザーは環境を意識せず同じ `refresh` コマンドを実行するだけ
- 将来 warden にビルトイン認証（LocalAuthentication、WebAuthn等）を追加しても command との互換性を維持可能
